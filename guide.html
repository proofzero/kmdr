kubelt guide

kubelt is a platform for containerizing data. With kubelt, users get the experience of Heroku with the scale of Kubernetes. Furthermore, kubelt provides a plugin architecture making it extensible into novel application models.

The goal of kubelt is to eliminate context switching by becoming the go-to platform and tool set for data scientists, data engineers and other data-centric roles. Context switching interrupts "flow" which leads to low productivity and high stress. With the data ecosystem growing in complexity and requiring ever more specialized tooling, context switching in working environments is becoming a larger problem.

Kubelt aims to improve the lifecycle of data-centric work with fast and compounding data sourcing and deployment models that keep users in flow.

How it works

kubelt is web3.0-first and is built on IPFS data substrates (public, managed, and on-prem) that distribute static and linked tree datasets as content-addressable data containers called "comets". Just like traditional containers, comet data containers are versioned and can be materialized for users and programs in a variety of ways.

Comet containers can also host arbitrary functions which can be composed alongside datasets as "comet flows" similar to ETL/ELT pipelines or APIs. When combined with schemas and types, comet containers and comet flows can be materialized and run anywhere with strong governance and validation, enabling novel applications.

Usage

Today, kubelt can be used via CLI. The CLI, called kmdr, works with a configurable daemon called ktrl to provide a "git-like" experience for end users.

kmdr and ktrl also provide a nice stable surface for "no/low-code" interfaces. kmdr and ktrl are both are open source.

KMDR CLI

The kmdr cli provides a set of commands for working with data containers. For example, here is a command to materialize a dataset to a user's local file system filtered by a predicate. (Note: predicates work against the kubelt schema and type system)

kmdr fetch acmecorp://comet:looney_tunes/dataset:road_runner_circuit_times['time < 1000', '&&', 'road == "hwy"']]@latest | kmdr csv -o road_runner_times.csv -

NOTE TBD: Materilzation mangement via package manager

From here the user can work with the data as a csv file, making changes and cleaning things up, then pushing it back at which time it will be assigned a new version.

kmdr csv -i road_runner_times.csv | kmdr push acmecorp://comet:looney_tunes/dataset:road_runner_circuit_times -

The user can also decide to run a program that yields a new dataset, and then push that new dataset to the data container.

python time_analysis.py # outputs new csv
kmdr csv -i road_runner_circuit_mean_times.csv | kmdr push acmecorp://comet:looney_tines/dataset:road_runner_circuit_mean_times

Best of all, the above steps can be piped into a single command with a new arbitrary transform function (more on that later). Contract/schema validations are also enforced between pipes.

kmdr build -e python mean_func team_analysis.py | kmdr push acmecorp://comet:looney_tunes/transform:mean_func


kmdr fetch acmecorp://comet:looney_tunes/road_runner_circuit_times['time < 1000', '&&', 'road == "hwy"']]@latest | \
kmdr run acmecorp://comet:looney_tunes/mean_func@0.1.0 | \
kmdr push acmecorp://comet:looney_tunes/road_runner_circuit_mean_times

To recap, let's break down the different components of the commands and arguments. In the above example, a space called acmecorp has been deployed along with a comet container called looney_tunes that contains a dataset called road_runner_circuit_times and a function that yields the mean of the road runner circuit times to a new dataset called road_runner_circuit_mean_times. And all of this is made available through crystal clear content addresses that follow this format:

kmdr fetch space://comet:container_name/dataset:dataset_name[:predicate]@version | \
kmdr run space://comet:container_name/transform:transform@version | \
kmdr push space://comet:container_name/dataset:dataset_name

This sequence of commands can also be described declaratively as a "comet flow" by using what is called a "kubelt manifest" and executed together with the following command:

kmdr run acmecorp://comet:looney_tunes/calculate_road_runner_circuit_mean | kmdr csv -o road_runner_circuit_mean_times.csv

NOTE: The commands listed above would also provide helpful validations and authorizations for the resources in question.

KTRL

ktrl is a local daemon that responds to kmdr commands for managing datasets, configurations, authentication, and local runtimes (wasm). The ktrl daemon is a subset of the remote kubelt cluster software that ensures local parity with all other members on the platform.

Addresses

Content on the kublet platform is stored as a hypergraph which allows kubelt content to be accessed directly via a node address or search path. For example,

A node address contains the name and optionally the type, version and disambiguator of the content in this format disambiguator:type:name@version

kmdr fetch road_runner_circuit_times # node address
kmdr fetch dataset:road_runner_circuit_times # node address if collision with type
kmdr fetch wileECoyote:dataset:road_runner_circuit_times # node address if collision with name
kmdr fetch wileECoyote:dataset:road_runner_circuit_times@0.1.0 # node address if version required

However, users may enjoy organizing content into hierarchies and accessing them through an entry point and search path analgous to the first example.

Note Disambiguators are aliases to the indvidual or group of keys used for auth.

Configuration

Behind the scenes end users or operators would have defined data containers through declarative "kubernetes-like" manifests or "configuration as code". These manifests are a set of APIs for ktrl and backend services to parse as objects and resources across the kubelt universe.

NOTE: Resources can be created directly through helper kmdr cli commands and saved as yaml via stdout for maintenance.

Here is the example above defined as kubelt manifests (sans governance/acl configurations):

---
apiVersion: kubelt.com/v1alpha1
kind: Space
metadata:
  name: acmecorp
spec:
  keys: []
---
apiVersion: kubelt.com/v1alpha1
kind: Comet
metadata:
  name: looney_tunes
  space: acmecorp
---
apiVersion: kubelt.com/v1alpha1
kind: Dataset
metadata:
  name: road_runner_times
  comet: looney_tunes 
spec:
  schema: road_runner_schema/lap_times
---
apiVersion: kubelt.com/v1alpha1
kind: Dataset
metadata:
  name: road_runner_mean_times
  comet: looney_tunes 
spec:
  schema: road_runner_schema/mean_times
---
apiVersion: kubelt.com/v1alpha1
kind: Transform
metadata:
  name: mean_func
  comet: looney_tunes 
spec:
  schema: 
    in: road_runner_schema/lap_times
    out: road_runner_schema/mean_times
---
apiVersion: kubelt.com/v1alpha1
kind: Schema
metadata:
  name: road_runner_schema
  space: acmecorp
spec:
  definitions: 
    lap_times: |
      circuit: string & ~= [a-zA-Z0-9_-]){3,20} // string that is alphanumeric between 3 and 20 characters in len
      lap: number & >0 & <100 // a lab number greater than 0 and less that 100
      time: number & >0 & <100000 // a number greater than 0 and less that 100000ms
    mean_times: |
      circuit: lap_times/circuit
      mean: lap_times/time
    mean_func_in: |
      lap_times: [ lap_times ]
    mean_func_out: |
      mean_times: [ mean_times ]
---
apiVersion: kubelt.com/v1alpha1
kind: CometFlow
metadata:
  name: calculate_road_runner_circuit_mean
  space: acmecorp
spec:
  jobs:
    fetch_times:
      steps:
      - name: fetch_circuit_times
        uses: kmdr/fetch
        with:
          source: looney_tunes/road_runner_times
          version: latest
          predicate:
            - 'time < 1000'
            - '&&'
            - 'road == "hwy"'
          # or with content address; uses: kmdr fetch acmecorp://comet:looney_tunes/road_runner_times['time < 1000', '&&', 'road == "hwy"']@latest
    
    calc_mean:
      steps:
      - name: calculate_circuit_mean
        needs: fetch_times
        uses: kmdr/run
        with:
          func: looney_tunes/calculate_road_runner_circuit_mean
          in: fetch_times.outputs.fetch_circuit_times

    push_mean:
      steps:
      - name: push_circuit_mean_times
        needs: calc_mean
        uses: kmdr/push
        with:
          in: calc_mean.outputs.calculate_circuit_mean
          sink: looney_tunes/road_runner_mean_times

Schemas and Types

Schemas and types unlock validation and data generation. Declaring types allows kubelt to transform flat, static files into trees of linked data that not only optimize storage but enable robust querying of datasets and pre/post validation of mutation events.

Schemas are standalone objects with their own content addresses that in the future can be accessed to generate test data and look alike data for DLP and fast data sourcing.

For example,

kmdr synth schema:road_runner_schema | kmdr csv -o road_runner_times.csv

Governance

Spaces and containers offer different levels of governance and access controls to kubelt objects and resources. Spaces are analogous to repositories and can be shared with multiple users and groups of users within and across organizational contexts. Comets are a subset of spaces that can be forked and subscribed to from one space to another (see advanced comet flows below for more).

Forking and Subscribing

Just like with git, kubelt offers users the ability to fork and subscribe to containers and other objects that users have privileged access to.

For instance, a user in the space acmecorp may choose to subscribe to the road_runner_circuit in their personal space named wileEcoyote in order to use it in his own analysis.

In addition to subscribing, wileEcoyote would fork the road_runner_schema to add new columns and types for a new dataset.

Subscribing and forking can be restricted to specific users or classes of users on the platform across contexts.

Advanced Comet Flows

One of the most powerful features of kubelt is users are able to bring the code to the data rather than moving the data to the code. This opens up possibilities for distributed machine learning and other classes of data science and data engineering typically restricted due to private and public defined compliance policies such as HIPPA, PCI, and others.

Contexts

Contexts are bound to specific deployments/clusters of kubelt, for instance public, managed/cloud, and on-prem. Users can have access to multiple contexts for cross-organizational collaboration (think hospitals, research institutions/universities, and enterprises).

Goverance can be enforced at the organizational level with private contexts.

Plugin Architecture

Similar to Kubernetes CRDs and kubectl cli, kubelt will provide the ability to publish and extend kubelt APIs and cli extensions with custom plugins to the open source kmdr and ktrl projects.

This plugin architecture is the same the core kubelt team uses to create commands like the csv command shown above.

Application Models

kubelt isn't just about data sourcing and data pipelines. Over time, kubelt can be levered into end-to-end application development. Some of our favourite application models that kubelt enables are below.

CQRS/ES and Reactive Applications

The Command Query Responsibility Segregation (CQRS) / Event Sourcing (ES) application model is a robust architecture for building resilient, scalable applications. The architecture pattern splits back ends into less frequent mutation services with reactive event streams and high frequency read services with highly cacheable APIs.

With CQRS/ES, every mutation is captured as an immutable event that can be replayed for triage, machine learning, and more in addition to forking new, fast readable databases for standalone APIs.

This is the default model for the kubelt platform however, at the moment we are focused on the write side first.

Embedded Applications

Since kubelt is built with IPFS, customers of kubelt can build distributed headless applications right into the OS of choice. For instance, we imagine a baked-in dropbox-like OS integration that materializes schema-driven files to file systems and distributes events, transforms and changes to all subscribers.

Bootstrapping

Spaces and comets can be watched and used to ship configurations, scripts and other objects to bootstrap environments like:

  • Severs
  • Developer machines
  • Edge compute
  • Side cars
  • ...and more

SDK

In addition to the kmdr cli, kubelt can use its protobuf definitions to quickly create polyglot SDKs for code-level interfacing to data containers with wrappers for popular libraries like pandas and numpy.
